## Verified Join Paths (9 Confirmed)

1. ✓ Sales Order → Order Items (1:N via salesOrder)
2. ✓ Order Items → Delivery Items (1:N via referenceSdDocument)
3. ✓ Delivery Items → Billing Items (1:N via referenceSdDocument)
4. ✓ Billing Headers → Items (1:N via billingDocument)
5. ✓ Billing Document → GL Entry (1:N via accountingDocument)
6. ✓ GL Entry → Payment (1:N via clearingAccountingDocument)
7. ✓ Sales Order → Customer (N:1 via soldToParty)
8. ✓ Order Item → Product (N:1 via material)
9. ✓ Delivery Item → Plant (N:1 via plant)

**Uncertain Join**: GL Entry ← referenceDocument (often NULL)
- **Mitigation**: Use GL-based linking via accountingDocument instead

## Risky Joins (5 Identified)

### 1. Journal Entry → Billing Document Reference (Often NULL)
- **Issue**: `referenceDocument` is NULL in many rows
- **Mitigation**: Use `accountingDocument` link from billing headers instead
- **Impact**: Medium (broken traces if not handled)

### 2. Datetime Inconsistency (Nested Time Struct)
- **Issue**: `creationTime` is `{hours, minutes, seconds}` object, not ISO-8601
- **Mitigation**: Flatten during ETL to full timestamp
- **Impact**: High (queries fail if not converted)

### 3. Zero-Padded String IDs (Not Numeric)
- **Issue**: `salesOrderItem="10"` must be compared as string, not number
- **Mitigation**: Type schema strictly as VARCHAR, never cast to INT
- **Impact**: Low (query returns no rows silently)

### 4. Sparse Product Availability (Dense, Actually)
- **Issue**: 3,036 product_plants records (69 products × 44 plants = dense)
- **Mitigation**: Safe to assume complete matrix; no NULL fallback needed
- **Impact**: Low (unexpected but handled by SQL)

### 5. Small Customer Base (8 Business Partners)
- **Issue**: Dataset is toy scenario (8 customers)
- **Mitigation**: Document assumption; focus on query power not volume
- **Impact**: Low (demo queries will be fast)
